1. Agentic AI란?

1. What is Agentic AI?

일반적인 LLM 사용은 "질문 → 답변"의 단발성 상호작용이다. Agentic AI는 이를 넘어서 LLM이 스스로 판단하고, 도구를 사용하고, 결과를 검증하는 자율적 루프를 구성한다.

Typical LLM usage is a one-shot "question → answer" interaction. Agentic AI goes beyond this — LLMs make decisions, use tools, and verify results in autonomous loops.

기존: 단일 LLM

Before: Single LLM

  • 사람이 프롬프트를 작성
  • Human writes the prompt
  • LLM이 한 번에 응답
  • LLM responds in one shot
  • 결과 검증은 사람 몫
  • Human verifies the result
  • 틀리면 다시 프롬프트 수정
  • If wrong, revise and re-prompt
  • 모든 작업에 같은 모델 사용
  • Same model for all tasks

Agentic: 에이전트 팀

Agentic: Agent Team

  • 사람은 목표만 제시
  • Human provides the goal only
  • 에이전트가 계획 → 실행 → 검증
  • Agents plan → execute → verify
  • 자동 품질 게이트 통과
  • Auto quality gate checks
  • 실패 시 에이전트가 재시도
  • Agents retry on failure
  • 작업별 최적 모델 라우팅
  • Route to optimal model per task
💡

핵심 전환: 사람이 "어떻게"를 지시하던 것에서, "무엇을"만 말하면 에이전트가 "어떻게"를 결정하는 구조로.

Key shift: from humans dictating "how" to agents deciding "how" — humans only specify "what."

3. 왜 멀티 티어인가?

3. Why Multi-Tier?

가장 강력한 모델 하나에 모든 작업을 맡기는 것이 최선일까? 실무에서는 세 가지 제약이 이를 막는다.

Is it best to give all tasks to the most powerful model? In practice, three constraints prevent this.

제약Constraint 문제Problem 해결Solution
비용 / 사용량 제한Cost / Rate Limits 프리미엄 모델의 API 호출 수, 토큰 한도 Premium model API call limits, token quotas 단순 작업은 저비용 모델에 위임 Delegate simple tasks to cheaper models
속도Speed 대형 모델은 응답이 느림 Large models respond slowly 경량 모델로 병렬 처리 Parallelize with lightweight models
적합성Fit 대형 모델이 단순 작업에서 오버엔지니어링 Large models over-engineer simple tasks 작업 복잡도에 맞는 모델 선택 Match model to task complexity

이 세 제약의 교차점에서 "역할별로 다른 모델을 쓰자"는 멀티 티어 아이디어가 탄생한다.

At the intersection of these constraints, the "use different models for different roles" multi-tier idea emerges.

에스컬레이션 원칙: 하위 tier 실패 시 반드시 한 단계씩 상승한다. T4에서 곧바로 T1으로 점프하는 것은 금지 — 중간 tier가 해결할 수 있는 문제를 최상위에 올리면 비용 낭비.

Escalation rule: On failure, escalate one tier at a time. Jumping from T4 directly to T1 is prohibited — it wastes the most expensive resource on problems mid-tiers could solve.

5. Blueprint: 에이전트 팀 설계도

5. Blueprint: Agent Team Design

Blueprint는 에이전트 팀의 구성과 워크플로우를 정의하는 YAML 파일이다. 사람이 목표를 말하면, 오케스트레이터가 자동으로 Blueprint를 생성한다.

A Blueprint is a YAML file defining the team composition and workflow. When you state a goal, the orchestrator auto-generates the Blueprint.

구성 요소Component 역할Purpose
에이전트 목록Agent List 누가 참여하는가 — 역할, 실행 위치(cloud/local), 도구 Who participates — role, execution target (cloud/local), tools
워크플로우Workflow 어떤 순서로 일하는가 — A의 결과가 B의 입력으로 In what order — A's output becomes B's input
에러 정책Error Policy 실패 시 어떻게 하는가 — 재시도, 에스컬레이션, 중단 What happens on failure — retry, escalate, abort
큐 메타데이터Queue Metadata 다른 Blueprint와 어떻게 병합하는가 How to merge with other Blueprints
제한Limits 에이전트 최대 3개, cloud 스텝 세션당 10개 Max 3 agents, 10 cloud steps per session
✅

Blueprint는 자동 Lint를 거친다. 워크플로우 연결 오류, 로컬 에이전트 프롬프트의 언어, cloud 비율 초과 등을 설계 직후 검사해 실행 전에 차단한다.

Blueprints pass auto-Lint. Workflow connection errors, local agent prompt language, cloud ratio violations — all checked immediately after design, blocked before execution.

Plan-Do-Review (Most Common)

소프트웨어 개발의 자연스러운 흐름을 반영한다. 계획 → 구현 → 리뷰를 각각 다른 에이전트가 담당해 한 에이전트의 편향을 다른 에이전트가 검증한다.

Mirrors the natural software development flow. Plan → Implement → Review, each handled by different agents so one agent's bias is checked by another.

7. 자동 품질 게이트

7. Automated Quality Gates

에이전트가 코드를 작성하면, 다른 에이전트가 자동으로 검증한다. 사람이 일일이 리뷰하지 않아도 기본적인 품질이 보장되는 구조다.

When an agent writes code, other agents automatically verify it. Basic quality is guaranteed without manual human review.

8. 세션 연속성: 에이전트의 기억

8. Session Continuity: Agent Memory

LLM 에이전트의 가장 큰 약점 중 하나는 세션 간 맥락 소실이다. 대화가 끝나면 모든 것을 잊는다. 이 문제를 구조적으로 해결하는 것이 핸드오프 시스템이다.

One of the biggest weaknesses of LLM agents is context loss between sessions. Everything is forgotten when the conversation ends. A handoff system solves this structurally.

세션 간 핸드오프와 지식 베이스 연동
Cross-session handoff with knowledge base integration
💡

"하지 말 것" 기록이 핵심이다. 에이전트는 과거의 실패를 기억하지 못하므로, 같은 실수를 반복할 위험이 있다. 핸드오프 파일에 실패한 접근법과 이유를 명시해 이를 방지한다.

The "do NOT retry" section is critical. Agents can't remember past failures, risking repeated mistakes. Handoff files explicitly record failed approaches and why, preventing this.
